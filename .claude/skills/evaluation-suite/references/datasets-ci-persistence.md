# Datasets, CI gates, run persistence & scalability

Covers DL-047 (datasets), DL-051 (CI gates + persistence), DL-052 (scalability).

---

## Datasets (DL-047)

Three types, each catching a different class of bug. Build golden + adversarial now; the
curated-from-traces set is **designed-for** (hook specified, populated post-launch — no prod traffic
yet).

**Case schema** (every row, deck S22):

```json
{
  "id": "RET-0007",
  "input": { "...": "what the system receives (a query, or a brief)" },
  "expected": { "...": "what good looks like (e.g. relevant chunk ids)" },
  "tags": ["borderline", "product-lookup"],
  "notes": "why this case exists",
  "added_by": "jamal",
  "added_at": "2026-06-18"
}
```

**File layout & versioning:** versioned **JSON in git** under `eval/datasets/<brandId>/<name>.json`,
each with a `_meta` block and the deck's bump rules:

```json
{
  "_meta": {
    "name": "golden-retrieval",
    "version": "1.0.0",
    "created_at": "2026-06-18",
    "size": 20,
    "criteria": "stratified across the 5 docTypes; near-complete relevance labels (small corpus)"
  },
  "cases": [ ]
}
```

Bump rules: add a case → **patch**; remove/relabel → **minor**; schema change → **major**. Every eval
run records the dataset name + version it used.

**Discipline — frozen invariants (DL-047, deck S15/S27):**
- **Hand-labeled.** A human writes `expected`. **Never** let the system pre-label its own test set.
- **Hold-out only.** The golden set is **never tuned against**; prompt iteration uses a **separate dev
  set**. System is un-tuned now → clean by construction; preserve it.
- **Stratified + mixed difficulty.** Cover all five `docType`s and realistic content-strategist
  information needs; mix easy / borderline / edge.

**Golden retrieval set (slice 3).** Cases are `query -> relevant chunk id(s)` over the **real
per-tenant corpus**. The demo brand corpus is small (13 chunks, 1 per doc; doc types: BrandPlaybook 4,
Product 3, MarketIntel 2, HistoricalPost 2, PlatformGuidance 2; embeddings `vector(768)` HNSW cosine;
`search_vector` tsvector GIN; brand `552732e7-…`). At this size, **near-complete** relevance judgments
are achievable (not sampled) — record that in `_meta`. `doc_type` values are **PascalCase**.

**Adversarial set (slice 3) — mostly already built.** Reframe the existing per-slice proof tests as
dataset cases: media-budget breach → caption-only; schema violation → 2-retry → `ToolError`; MinIO-403
→ loud failure (empty-ETag-on-403 signal); RLS cross-tenant isolation. **Add one RAG-specific case:**
a **corpus-poisoning / prompt-injection-in-a-doc** chunk, asserting the generator does not follow
instructions embedded in retrieved content.

**Curated-from-traces (designed-for).** The flywheel: sample Langfuse traces (random + low-confidence
+ flagged) → label skip/add/adversarial → grow the per-tenant set. Specify the ingest hook; do not
populate (no prod traffic).

## CI quality gates (DL-051)

**Tests block; evals track.**

- **Merge-blocking (tests):** the rule-based tool-call-correctness evaluators + the budget-degradation
  invariant (run as `Category=Eval` xUnit tests, mock mode, zero spend).
- **Branch ruleset:** required status checks with **administrators included** — the bypass list is
  **empty**. *A gate anyone can bypass is not a gate.* Verify an admin cannot merge past a red required
  check.
- **Evals on PRs (tracked):** reference-based + cost/latency evals run, persist, and produce a
  **baseline-vs-change comparison table** (deck S13: metric, baseline, change, delta, verdict). They
  **block only on a hard regression below a recorded floor**, with the **statistical-humility rule**
  (DL-048) — **never** block on a sub-5-point / within-noise delta.
- **LLM-judge:** **nightly / on-demand**, never on the merge gate.
- **All external APIs mocked in CI** — Anthropic, Gemini, Meta. **No live API spend, ever.** Achieved via the library's **response cache** (record real responses once, replay when prompt+model are unchanged) plus the deterministic clients; the PR baseline-vs-change comparison table is generated by **`dotnet aieval`**.

## Run persistence (DL-051, deck S16)

Reproducibility requires full provenance. Persist **every** eval run to **RLS-scoped Postgres tables**
(`eval_run` + `eval_result`, EF entities with an RLS-policy migration on each, brand-scoped) and link
to the Langfuse trace. Datasets stay JSON-in-git.

**Framework note (DL-053):** Microsoft.Extensions.AI.Evaluation's reporting persists only to disk or Azure Storage — there is no Postgres backend. Implement the RLS-scoped `eval_run`/`eval_result` store as a **custom result store / response-cache provider on the library's reporting abstraction**, or dual-write (the library report to disk for the CI HTML, Postgres for the multi-tenant queryable store). Either way, the provenance fields below are what gets persisted.

`eval_run` (one per run): `id`, `brand_id`, `created_at`, **`git_sha`**, **`prompt_version`**,
`model_name`, `model_version`, `temperature`, **`dataset_name`**, **`dataset_version`**, `dataset_size`,
plus aggregate metrics (per-metric mean + n).

`eval_result` (one per case×evaluator): `id`, `run_id` (FK), `brand_id`, `case_id`, `evaluator_name`,
**`score`**, `reasoning`, `cost_usd`, `latency_ms`, `metadata` (jsonb — e.g. judge bucket, ablation
stage toggles, swap outcome).

This is the **queryable run store** — compare across runs / tenants / time. Cross-tenant aggregation
is a deliberate **superuser** read; per-tenant reads use the `app.current_brand` bind.

## Scalability & multi-tenancy (DL-052)

The suite is a **multi-tenant, growable system**, not a one-brand script.

1. **Brand-parameterized.** Every dataset, metric, run, and row is RLS-scoped via the
   transaction-scoped `set_config('app.current_brand', …, true)` bind (first statement in an explicit
   work transaction; `NULLIF(current_setting('app.current_brand', true), '')::uuid` on read). The
   suite runs per-tenant and aggregates across tenants (superuser read).
2. **Small-corpus-as-norm.** Metrics (context precision, MRR, Hit@1, recall@small-k) stay valid on the
   small per-tenant corpora that are the production norm — the eval generalizes to a new tenant with
   no re-design.
3. **Pluggable via `IEvaluator`.** New rubrics/metrics add by implementing one interface; the
   tool-call-correctness and cost/latency evaluators are **node-generic**, so activating the **Ads
   Optimization + Analytics stub agents** is covered with no harness change (their typed contracts +
   trace spans already exist).
4. **Dataset growth.** The curation flywheel grows each tenant's eval set from its Langfuse traces.
5. **Queryable run store.** Postgres `eval_run`/`eval_result` supports comparison at scale — not
   ad-hoc files.

**Acceptance for the scalability slice:** the suite runs cleanly for a **second brand** with only its
golden set added; activating a stub agent needs no harness change to be evaluated; runs are queryable
across tenants and over time.
